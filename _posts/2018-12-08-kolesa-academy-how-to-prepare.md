# Фундаметельные знания
Возможно вы уже разрабатываете базовые Android приложении, это хорошо.
Но без фундаментальных знании вы современем уткнетесь к том,
что будете писать плохой код и не знать как правильно организовывать код.
Чтобы решить этот вопрос, вам нужно изучить больше про концепцию
Object-Oriented Programming и базовые прицнипы программирования,
которые как раз преподаются в университетсих курсах по Computer Science.

## Принципы OOP
Прицип OOP включает себя: Encapsulation, Abstraction, Inheritance и Polymorphism.
Этот принцип для начинающих разработчиков возможно покажутся страшными словами.
Поэтому попробуем разобраться в каждом из них.

### Encapsulation (Инкапсуляция)
Когда вы водите машину, вы нажимаете на газ чтобы ускорить,
управляете рулем чтобы повернуть, отпускаете газ и нажимаете на тормоза
чтобы остановить. Все эти детали кроется подкапотом машины, которые разработали
инженеры. Вам как водителю предоставлен только интерфейс для того чтобы могли
комфорно водить. Эти публичные части машины можно рассмотреть как публичные части
объекта, достыпные нам объекту водитель.

Инкапсуция - возможность объекта скрывать детали реализации от других объектов,
давая при этом публичные либо частично скрытые доступ к нему.
В случае с языком Java предоставляется возможность объявлять переменные или методы
как public, private или protected, если видимость доступен только для подкласса.

Если вы закрываете(private) состояние объекта, предоставляя при этом возможность
менять состояние через публичные(public) методы, то вы успешно применяете
принцип инкапсуляции.

### Abstraction (Абстракция)
Когда вы разрабатываете программу, со временем проект становится очень большим и сложным.
Различные объекты общаются друг с другом часто. Поэтому поддерживать такие проекты с
годами становится сложнее - требование постоянно меняются. Концепт "абстракция"
как раз целится на то чтобы упростить эту задачу.

Абстракция - это модель некого объекта из реальных миров либо некое назначение в проекте.
Данный модель должен предоставить другим объектам **только** механизм взаймодействии
связанное с данным моделью.

Абстракцию можно рассмотривать как расширение инкапсуляции.
Кстати, поэтому абстракцию зачастую не включают в общий список принципов OOP.

Представьте себе что вам нужно создать объект кофе машинка. Пользователь должен
предоставить чашку чая и нажать на кнопку чтобы кофе машинка начала наполнять чашку кофе.
Как будет изготовливать кофе знать не должен - пользователь только взайодействует с
публичными методами кофе машинки. Желательно чтобы этот абстрактный объект был
просто для пользования и неизменялся интерфейс для обращения к нему.

### Inheritance (Наследование)
Часто когда пишите программу, некоторые объекты имеют общие логики,
но они не совсем одинаковые сущности.

Например, кошка и собака являются животными.
Хорошо бы выделить общую логику свойства животных, как, например, способность кушать.
Мы бы могли создать родительского класса Animal для Cat и Dog - это и есть Наследование.

Но важно отметить, что не все животные имеют четыре лапы как кошка и собака.
Отрастим ножки для кита, чтобы тоже умел ходить в наших программах?

![whale with legs](/assets/academy/whale-with-legs.png)

Такие свойства, как умение ходить или плавать можно выделить в interface.
Но при этом класс Dog может научиться плавать.
Язык программирования Java позволяет наследоваться только от одного класса,
но есть возмоность реализовать множество interface.

```java
interface FourLegged {
  public void run(Point destination);
}
interface Swimmers {
  public void swim(Point destination);
}
class Dog extends Animal implements FourLegged, Swimmers {
  ...
}
```

### Polymorphism (Полиморфизм)
В принципе наследования мы рассмотрели, что четвероногие животные умеют ходить.
Но каждые из них могут ходить по разному.

Принцип Полиформизм описывает возможность наследников реализовывать общие операции по своему.
Так, например, мы можем реализовать метод run() немного по разному зависимости от семейства животных.

Если бы в программе содержалось список животных, то пробегаясь по этому списку,
каждый объект в этом списке выполнял бы операцию run() по разному.

```java
private void runAll(List<FourLegged> fourLeggeds, Point point) {
  for (FourLegged fourLegged : fourLeggeds) {
    fourLegged.run(point);
  }
}
```

## Design Pattern (Паттерны проектирования)
Вы можете успешно писать Android приложении, используя известные библиотеки для
решения определенных задач или копируя готовые функции для решения задачи.
Но такой подход разработки со временем приведет вас к тому,
что оптимазация функции или замена существующих библиотек будет очень затруднительным.
Паттерны проектирования как раз помогает вам, чтобы вы могли решить часто встречающие такие проблемы.

Паттерны иногда сложно применять в реальности, так как у паттерна нет определенного кода.
В описании паттерна приводится только общая концепция для решения той или иной проблемы.
Есть, конечно, примеры реализации, но зачастую они не подходит под ваш проект.
Поэтому приходится еще подстраивать паттерны по разному в зависимости от нужд вашей программы.
А многие иногда даже не представляют,
что их решение проблемы включает в себя определенную концепцию паттерна.

### Зачем же изучать паттерны
В чем отличие, например, профессионального ремесленника от новичков?
Профессионал знает как применить те или иные инструменты под различных условиях.

В разработке Android приложении то же самое.

1. Зачастую многие проблемы можно решить, применив готовые решения,
которые как раз описываются в концепциях паттерна.
Вы можете сами додуматься до такого решения, но зачем изобретать велосипед?
2. Структура классов паттерна унифицирован,
поэтому сложнее ошибиться при проектирование программ.
Вы действуете по шаблону, которое описаны в паттернах.
3. Как объяснить другому разработчику, как вы подошли к решению проблемы в проекте -
"Я создал класс X, Y, Z и применил немного магии"?
Проще будет если вы скажете, что применили паттерн под названием X.

### Классификация паттернов
Многие паттерны были открыты умными дядьками программистами за долго нас.
Со временем их количество возрасло десятками, что потребовалось их классифицировать,
чтобы легче было найти нужные паттерны для решения общих проблем:

1. Creational patterns (Порождающие паттерны) - они имеют дело с механизмами создания объектов,
подходящие для разных ситуациях. В этом классе включается: Abstract Factory,
Builder, Factory Method, Object Pool, Prototype и Singleton.
Некоторыми из них вы наверника уже сталкивались.
2. Structural patterns (Структурные паттерны) - данный класс предназначен для
упрощения реализации связи между объектами. Мы каждый день применяем такие паттерны,
как Adapter, Decorator, Facade в рабочих проектах. Так же есть Bridge, Composite,
Flyweight, Private Data Class, Proxy.
3. Behavioral patterns (Поведенческие паттерны) - эти паттерны увеличивают гибкость
в осуществлении коммуникации между объектами. Например, многим известный Observer
относится к данному классу паттеранов.

Рекомендую зайти на сайт [refactoring.guru](https://refactoring.guru/ru/design-patterns)
чтобы изучить грубже паттерны проектирования.

## Алгоритмы и структуры данных
Во время собеседования мы спрашиваем вопросы связанное с алгоритмом и структурой данный.
Некоторые кандидаты говорят нам, что они не знают как решить,
так как при разработке мобильных приложении, они не решают подобные задачи.
Нужно ли знать рарботчику о том, как выполняется сортировка данных при вызове
функции Collections.sort или как лучше всего хранить данные,
которые получаем от внешного источника?

Можно взять реальные примеры из жизни рабработчика.
Например, мы получаем по API данные для отображения списка комментариев на объявление.
Нам приходят JSON Array без каких либо вложенности,
есть только идентификатор комментария, на которую пользователь отвечает.
Как организовать структуру данных так, чтобы при отображение могли вывести виде
дерева и при этом оптимальнее всего было искать нужный объект.

Хороший разработчик должен изучать их, чтобы решать задачи оптимально.
Без знания алгоритмов, при решение задачи, вы попробуете решить своим путем,
что зачастую окажется не эффективным при вополнение кода.
Без знания структуры данных, вы затруднитесь выбирать нужные структуры данных.
Что использовать при создание объекта данных - банально через List или Map?
Если у вы не знаете как они работает при записи и чтения,
не сможете быстро решить, что оптимальнее всего для вашей задачи.

Программирование - это все об алгоритмах и структурах данных.
Если у вас имеется базовые знания структуры данных или алгоритмов,
вы бы уже знали решение проблемы, до того как напишите строчку кода.
Таким образом мы заранее можем понять эффективность разработчика.

### Опытным разработчикам тоже тяжело пройти интервию
Независимо от опыта разработчиков,
приходится готовиться к вопросам по алгоритмам во время интервью.
С годами теряется знание базовых алгоритмов, так как естественно мозг забывает об этом.
Но если вы помните базовые принципы выполнения алгоритмов, то легче вспомнить его при повторение.

Возможно во время выполнения задачи по алгоритмам, вы столкнетесь с проблемами,
которые не решали, но достаточно знать базовые понимания алгоритмов и структуры данных.
Тем более мы не спрашиваем задачи уровня Google - достаточно простые задачи,
которые можно встретить начальных уровнях в сборниках задач.
